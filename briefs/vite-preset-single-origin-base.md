# Бриф: `@omnifield/vite-preset` — гибкий frontend-пресет с single-origin base

> **Трек:** Foundation — Шаг 5, инфра-достройка (frontend через дверь, builder-agnostic)
> **Адресат:** архитектор / owner **devopser** (зона `packages/` — новый пресет; owner-skeleton)
> **Заказчик:** workspace-архитектор (omnifield-hub)

## North star (важнейшее — read first)
Frontend-настройки скелета — **гибкие, конфиг/плагин-driven**. Сейчас канонный **vite** (нативка), но
**скоро — свой веб-фреймворк**, на который переведём все фронты: кастом-мехи по большей части через **плагины**.
Поэтому: **ноль хардкода vite-специфики в продукте**; продукт расширяет пресет; будущий фреймворк = плагин/
вариант пресета, НЕ переписывание. Single-origin base — **из манифеста (единый источник), builder-agnostic**.

## Зачем (факт с живой двери)
Дверь Шага 5.3 монтирует фронт под `:8080/<name>`, но vite отдаёт SPA под `/` (base по умолчанию) →
`:8080/chater` = 404. Каждый фронт под своим route-префиксом обязан иметь **base = свой манифест-route**
(`/<name>/`). Хардкодить `base` в каждый `vite.config` = не гибко (и не переживёт миграцию на фреймворк).

## Скоуп (зона devopser/packages)
1. **`@omnifield/vite-preset`** (паттерн `biome-preset`/`nx-preset` — публикуемый пакет, README, продукты
   расширяют). Фабрика (напр. `defineOmnifieldVite(opts)`) → vite-конфиг:
   - **base из манифеста (единый источник):** префикс = front-route продукта из его `omnifield.yaml`
     (`reach.routes` под `/<name>`). Как достаётся (генерён в конфиг при init/synk, читается из env, или
     хелпер) — на дизайн owner'а; **не хардкод per-product**, не дубль контракта.
   - **host:0.0.0.0 / dev-server** канон (G1), allowedHosts, и прочий общий vite-канон — в пресете.
   - **Плагин-расширяемость:** продукт передаёт свои плагины (solid и т.п.) через opts; пресет их
     компонует. Будущий фреймворк подключается тем же слотом (плагин) / вариантом пресета — **гибкость
     заложить сейчас**, чтобы миграция была конфиг, не rewrite.
2. **Дока пресета** — как продукт-фронт его расширяет (5 строк `vite.config`), где берётся base, куда
   вставлять плагины (место под будущий фреймворк).

## Handoff → chater (первый потребитель)
- `web/vite.config.ts` расширяет `@omnifield/vite-preset` (base `/chater/` из манифеста, ноль хардкода).
- **API-клиент фронта через door-контракт:** сейчас зовёт `/chater/rooms` (vite-proxy→backend); под
  single-origin звать `/api/chater/...` (door-маршрут backend). Реконсиляция dev↔door — chater-app зона.
- Живой прог: `:8080/chater` → SPA (200), API-вызовы доходят до backend через `/api/chater`.

## Вне скоупа / связь
- Публикация пакета в реестр — как biome/nx-preset (зона, что и они). Если bundle/vendor — по образцу
  contract-manifest в hub-core.
- Weber-фронт (`/sandbox`|`/weber`) — тот же пресет позже (обобщение канона).
- Мелочь-флаг (не сюда): devbox-services probe-timeout 30s мал для cold `go run` (backend поднимается >30s,
  ложная ⚠) — bump probe / go-build-cache переживает recreate. Отдельно.
