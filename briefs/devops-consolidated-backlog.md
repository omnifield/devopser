### ⚠️ Контр-проект по §1.3 — registry-npm, возможно, вообще не стек

Столкновение двух канонов, которое Verdaccio-на-dev-хосте не разрешает:
«clone → install → работает» (P0) × «машина = cattle». Storage Verdaccio живёт
в volume ОДНОЙ машины → на второй тачке (ровно наш workstation-сценарий!)
`pnpm install` с `@omnifield/*`-пресетами падает — registry там пустой. Синк
volume'а между машинами = костыль (POLICY запрещает).

Варианты:
- **(а) GitHub Packages npm-registry — рекомендую.** НЕ новый вендор: GitHub уже
  держит сами репо (без него нет и clone), auth тем же токеном (пост-шаг git-auth
  уже в workstation/README), доступен с любой машины, ноль поднимаемой инфры.
  Прецедент «Nx Cloud отвергнут» не нарушается — там добавлялся новый внешний
  сервис, здесь нет. Lock-in нулевой: переезд на self-host позже = строка в
  `.npmrc` + republish.
- (б) Verdaccio-стек на dev-хосте — честен только при явном решении «экосистема
  однохостовая до VPS»; вторая тачка репо не соберёт.
- (в) Отложить D2 до VPS-провайдера — просто поздно, D2 нужен раньше.

При любом варианте Verdaccio остаётся в инструментарии как эфемерный
`nx local-registry` для локальной проверки publish-флоу — это не стек и не бэклог.
**Нужна резолюция оракула/user.** Если (а) — п.3 бэклога растворяется внутрь D2
(строка в пресетах + doc), отдельного деливерабла нет.
