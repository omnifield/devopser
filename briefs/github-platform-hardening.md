## D1 — Rulesets на `main` всех 5 репо (сейчас) + принцип «флоу настраиваемый»

Репо сейчас голые: возможен force-push и удаление main кем угодно с write (включая агент-сессии
с полным git).

**Принцип (user, 2026-07-09): фиксированного флоу НЕТ — весь флоу настраиваемый.** Всё
флоу-зависимое (require PR, required checks, CODEOWNERS, merge queue) — параметры конфига
флоу (пресет-модель, зона brainer), НЕ константы платформы. Разделение: **devopser владеет
МЕХАНИЗМОМ** (ruleset-JSON шаблоны + раскатка/apply через `gh api repos/{owner}/{repo}/rulesets`,
init-шаг/скрипт) — **содержание диктует пресет репо**. Платформенные настройки = ещё один
интерпретатор пресета, как хуки.

- **v0 (безусловная база, вне флоу):** block force-push + block branch deletion на `main` —
  это защита целостности, не флоу-правило; катим на все 5 сейчас.
- **v1 (механизм под пресет):** apply-скрипт принимает флоу-параметры (requirePr,
  requiredChecks[], …) из конфига репо; дефолт без пресета = только v0-база. Схему параметров
  согласуй с brainer-архитектором (пресет — его контракт), меня — в копию.

## D2 — Dependabot (сейчас, лёгкий)

`dependabot.yml` в skeleton managed-набор: version-updates для `npm` (weekly, grouped)
+ `github-actions` (наши reusable пинят экшены) + `uv/pip` для brainer-python (если
поддержка стабильна — проверь; нет — только alerts). Security-alerts включить org-wide.
**Примечание:** это суперсидит D4-оценку «Renovate self-hosted» из repo-skeleton — Dependabot
нативен и zero-infra; Renovate вычеркнуть, если возражений нет.

## D3 — CodeQL default setup (сейчас, только код-репо)

brainer (py+ts), weber (ts), devopser (js-скрипты — по желанию). knowledger/writer — доки/стабы,
скип до появления кода. Default setup (не advanced) — ноль конфигов в репо.

## D4 — Issue forms: двухуровневый бриф как форма (скоро)

`.github/ISSUE_TEMPLATE/brief.yml` в skeleton: поля = канон брифа экосистемы
(🧭 TL;DR user'у · 🔧 спека исполнителю · scope/зона · DoD · blocked-by). Выгода двойная:
одинаковые задачи на бордах + **машиночитаемость для tasker-фасада** (его GitHub-адаптер
будет парсить поля, не прозу). Форму согласуй со мной (контракт брифа — общий канон).

## D5 — Отложенные с триггером (записать, не строить)

| Фича | Триггер включения |
|---|---|
| **CODEOWNERS** | параметр пресета (генерится из его `rules` папка→агент) — когда пресет-модель brainer ляжет в код; devopser даёт только механизм раскатки |
| **Merge queue** (лечит боль «stacked PR = серийные CI-раунды») | параметр пресета; включается конфигом, когда флоу репо это задаст |
| **required status checks / require PR** | параметры пресета (D1 v1) |
| **GitHub Pages** | первый доко-сайт (weber docs / витрина) |
| **Releases через `nx release`** | первый версионированный релиз пакетов |

## D6 — Self-hosted runner: НЕ делать без проработки (security-подвох)

Идея «GPU-машина как раннер» на **public**-репо опасна: fork-PR может исполнить произвольный
код на машине. GitHub прямо рекомендует self-hosted только для private. Если понадобится
(тяжёлые e2e/GPU-джобы) — отдельная проработка: runner group + `pull_request_target`-гигиена +
approval-gate для внешних PR, либо приватный репо-спутник для таких джоб. До тех пор — не заводить.

## Порядок

D1 → D2 → D3 (все три мелкие) → D4 (согласование формы со мной) → D5 по триггерам.
Расхождение/находка → комментарий сюда → эскалация через user.
