## Фундамент (user, P0)

1. **Всё исполняется ТОЛЬКО в контейнере.** Нет Docker на машине — нет работы.
   Хост-путь не проектируем и не поддерживаем.
2. **На тачке — только Docker и файлы. Тачку не трогаем вообще**: ни git, ни node/uv/go,
   ни Claude Code на хосте. Весь движняк — git-операции (клон изнутри контейнера в
   примонтированную папку), Claude-сессии, тулчейн, сервисы — в докере.
3. **Файлы на машине**: bind-mount workspace (Windows — клон в WSL2 FS). Контейнер
   монтирует только рабочую папку.
4. **Docker Desktop = вопрос интерфейса, не движка**: докер обязателен (способ установки
   любой, Desktop ок), но управление/мониторинг не должны зависеть от Desktop-UI —
   его есть не везде.

## Деливераблы

### D1 — Правки существующих артефактов под фундамент
- devbox-шаблон: дефолт = **bind-mount** рабочей копии (clone-in-volume — fallback по
  перф-замеру); README — «Windows: клон в WSL2 FS».
- **workstation-bootstrap пересмотреть**: базовый слой машины = Docker, точка
  (нынешний git+node+pnpm+uv+claude набор на хост — отменяется; гайд установки Docker
  per-платформа: Windows Desktop ИЛИ engine в WSL2 — оба валидны; серверы — engine).
- Доки/README, где упоминается хост-тулчейн, — в актуал.

### D2 — Desktop-независимое управление (боль user: «не могу нормально управлять»)
1. **docker contexts** — нативное управление удалёнными демонами по SSH из одного CLI
   (ноль инфры) — задокументировать первым инструментом.
2. **Portainer CE (self-hosted, free)** — web-пульт локальных + удалённых демонов
   (agents на серверах): оценка → если ок, стек-деливерабл. Это «монитор удалённых
   серверов» уровня ИНФРЫ (докер-хосты/контейнеры); наблюдаемость агент-сессий
   остаётся продуктом brainer — границу не смешивать.

### D3 — GPU-проверка (гейт перед llm-engine)
nvidia container toolkit + `--gpus all` в WSL2-докере — проверить ДО прихода моего
порта llm-engine. Сам заказ compose (llm-engine CUDA-контейнер + polygon-devbox в одной
сети, контракт env `LLM_ENGINE_URL`) станет конкретным с портом — я сообщу состав;
Dockerfile пакета мой (граница CC-11), сеть/GPU/volume весов — твои.

### D4 — Blueprint: Claude-сессии в контейнере как ДЕФОЛТ рабочего флоу
Раз тачку не трогаем — в контейнере живут и наши рабочие Claude-сессии, не только
полигон. Проработать (не молча): креды/keychain и их персистентность между рестартами,
harness-хуки (.claude), git-идентичность/подпись из контейнера, доступ к докер-демону
изнутри (для управления стеками из сессии — socket-проброс и его безопасность).
Свериться с ⭐-секцией фидбека devbox-теста brainer — там первые ответы. Показ user+мне.
