# Brief — hub-core как хаб-сервис под файловой изоляцией (discovery через Канал)

| | |
|---|---|
| **Адресат** | devopser (зона `hub-core` + `stacks/gateway`) — запускает user |
| **От** | workspace-архитектор (в `omnifield-hub`), 2026-07-13 |
| **Основание** | топология изменилась ПОСЛЕ `hub-core-design.md`: продукты стали **файлово-изолированы** (каждый devbox монтирует только свой репо), и появился **реальный тонкий хаб-контейнер `omnifield-hub` с docker-socket (Канал)**. | **Класс** | достройка инк2 под новую топологию (НЕ редизайн; канон дизайна сохраняется) |
| **Статус** | на исполнение owner'ом |

## Проблема (одна, конкретная)
`generate.mjs` строит реестр **сканируя сиблинг-папки `/workspaces`** и читая их `omnifield.yaml`.
В монолите god-mount видел все репо — работало. Сейчас devopser (и любой продукт) монтирует **только
свой репо** → прогон в devopser даёт `✖ ни одного валидного манифеста — нечего генерить`.
**Gateway-конфиг больше нечем регенерировать**; текущий рабочий `nginx.conf` (маршруты brainer+weber) —
застывший артефакт эпохи монолита.

Корень: fs-скан сиблингов держится на файловой НЕ-изоляции (god-mount), которую мы намеренно убрали.
Дизайн уже даёт ответ, как чинить: **хаб знает про все контейнеры и говорит с ними Каналом** — значит
манифесты надо собирать **Каналом**, а не с файловой системы.

## Цель
Сделать hub-core **реально исполняемым как хаб-сервис в `omnifield-hub`** (у него есть socket/Канал,
node/pnpm; репо продуктов он НЕ монтирует), заменив fs-скан на **сбор манифестов через Канал**.
Реализация §6/§7 дизайна в изолированной топологии. Ядро остаётся тонким: **только Реестр + Дверь**.

## Скоуп (зона devopser/hub-core)
1. **Discovery через Канал** вместо fs-скана сиблингов: перечислить продукт-контейнеры на docker-сети
   `omnifield-gateway` → достать из каждого его `omnifield.yaml` (docker exec/cp) → собрать реестр.
   - Источник правды = `omnifield.yaml` в корне продукт-репо (блюпринт §4). Не менять контракт.
   - Отсутствует/невалиден манифест = продукт не в реестре, **loud-warn** (сохранить Р2/§2, не тихо).
   - Alias продукта на сети (= имя) уже = имя репо = `manifest.name` — использовать как ключ.
2. **Исполнимость в хабе:** hub-core должен запускаться в `omnifield-hub`, который **не монтирует
   код продуктов**. Оформить так, чтобы ядро жило как хаб-сервис (образ/пакет/entrypoint — на выбор
   owner'а), а не как «скрипт, руками запускаемый в devopser».
   - **Развязать dep `@omnifield/contract-manifest`:** сейчас `file:../../knowledger/contracts/manifest`
     — под изоляцией из хаба не резолвится. Пакет уже собран (`dist/`) и `private:false` под публикацию.
     Способ (публикация в реестр / бандл в образ hub-core) — решение owner'а. ⚠️ Если нужна **публикация**
     пакета — это действие **knowledger** (его зона), координируется отдельным хендоффом, НЕ в этом DoD.
3. **Доставка gateway-конфига** без хостового coupling'а: сейчас gateway биндит
   `/home/egorr/omnifield/devopser/stacks/gateway/{nginx.conf,hub}` (хостовый путь devopser). При
   генераторе, исполняемом в хабе, определить доставку сгенерённого `nginx.conf` + лендинга в nginx-
   контейнер (общий volume, либо запись через Канал). Сохранить single-origin (наружу только `:8080`).
4. Сохранить `--check` дрейф-гейт (шаг CI «регенери»). Ноль рукописных строк в nginx/лендинге.

## Проверка (DoD)
- Из `omnifield-hub` одна команда регенерит маршруты gateway + лендинг для **всех продуктов с валидным
  `omnifield.yaml`** (сейчас brainer, weber), источник — **Канал**, без единого допущения о соседних
  папках `/workspaces`.
- Погашенный/отсутствующий продукт-манифест → loud-warn, продукт просто не в реестре (не падение).
- Gateway отдаёт регенерённый конфиг; `curl -sI localhost:8080/brainer/` и `/weber` → живой маршрут.
- `generate --check` ловит дрейф. Наружу машины по-прежнему торчит только `omnifield-gateway :8080`.

## Границы / не-цели
- **Провижининг флота (docker run devbox'ов) — НЕ сюда.** По Р1 ops/управление контейнерами = Канал
  (Portainer на старте, свой control-plane позже). generate.mjs остаётся тонким (реестр+дверь).
- **Portainer как Канал** — отдельный заход (Р1), не в этом брифе.
- **Продукты без манифеста** (chater/knowledger/writer/devopser) просто не в реестре — добавление их
  `omnifield.yaml` = действие каждого owner'а, не DoD devopser'а (можно параллельным хендоффом).
- **Публикация contract-manifest** (если выбран этот путь) = зона knowledger, отдельный хендофф.
- Порты/маршруты продуктов = контракт (`registry/ports.md`) — не менять, только регенерить корректно.

## Контекст-факты (проверены через Канал 2026-07-13)
- `omnifield-hub`: маунты = socket + `omnifield-secrets` + `omnifield-docs`; на сети `omnifield-gateway`
  alias=hub; образ = `ghcr.io/omnifield/devbox` (node/pnpm есть). Socket работает под vscode (группа
  `dockerhost`). Репо продуктов НЕ монтирует.
- Манифесты есть только у **brainer** и **weber**; у devopser/chater/knowledger/writer — нет.
- `@omnifield/contract-manifest@0.1.0` собран (`dist/` + `omnifield.schema.json`), `private:false`.
- gateway (nginx) биндит конфиг с хостового пути devopser (см. §3).
