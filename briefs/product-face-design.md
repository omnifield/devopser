## 0. Что такое лицо
**У каждого продукта есть ЛИЦО** — то, что открывается по его reach-роуту в хабе (превью → лицо).
Два вида:
- **интерактивный** (brainer) → лицо = **его собственный UI** (dev-сервер за роутом). Готово, работа не нужна.
- **пакетный** (weber — фреймворк `@weber/*`, юзеру тыкать нечего) → лицо = **сгенерённое описание пакетов**
  (что за пакеты, что делают), НЕ sandbox-полигон.

## 1. Развилка КОНТРАКТА (knowledger) — как хаб узнаёт вид лица
Манифест `type` (frontend/backend/fullstack/service) НЕ отличает «интерактивный» от «пакетного»
(weber — фреймворк, ни один тип не подходит; и у него ЕСТЬ dev-сервер sandbox, который НЕ лицо —
инференс по «есть dev-сервер» врёт). Значит манифесту нужно **явно выразить вид лица**. Варианты:
- **(1a)** новый `type`: `library`/`package` (или `framework`) — пакетные продукты;
- **(1b)** отдельное поле `face: "app" | "packages"` (или `interactive|generated`).
*Рекомендация:* **(1b)** — `face` ортогонален `type` (пакетный бэкенд тоже возможен), тоньше не ломает
enum. Это **заказ knowledger** (правка `@omnifield/contract-manifest`, тонкая) — пришлю через git.

## 2. Развилка ГЕНЕРАЦИИ — кто строит и отдаёт пакетное лицо
Источник контента лицам-описания есть уже сейчас: `package.json` описания `@weber/*` + README + манифест.
Кто рендерит и отдаёт страницу на роуте:
- **(A) hub-core генерит + хаб отдаёт** пакетное лицо (сканит `package.json` пакетов продукта →
  описание-страница, статикой за gateway/хабом на reach-роуте). **Тонко, БЕЗ контейнера на продукт.**
  «Часть шаблона» = скелет-конвенция «у пакетов есть описания» (линт). ← рекомендация.
- **(B) шаблон продукта несёт face-app** (свой минимальный сервер/страница). Продукт-owned, но
  +контейнер на каждый пакетный продукт (лишний вес).

*Рекомендация:* **(A)** — thin, никаких лишних контейнеров, хаб рендерит из метаданных, что у продукта
уже есть. Интерактивные лица (A не трогает) = собственный UI продукта за роутом.

## 3. weber — первый пакетный кейс (DoD)
- weber-манифест: `face: packages` (после contract-правки §1).
- hub-core: для `face: packages` сканит `@weber/*` (`core/`, `packages/{style,ui,state,…}`) →
  генерит описание-страницу (пакет · что делает) → отдаёт на `/weber`.
- **DoD:** `:8080/weber` показывает **сгенерённое описание пакетов weber** (не sandbox); brainer
  `/brainer` = его UI. У каждого продукта — лицо на его reach-роуте. Ни строки лица руками.

## 4. Границы
- Лицо интерактивного продукта = его UI, хаб только линкует (не рендерит). Генерим только пакетные.
- Sandbox-app'ы weber (`apps/sandbox`) = внутренние тест-полигоны, в хаб не выносятся.
- Контент пакетного лица = из метаданных продукта (описания пакетов) — источник у продукта, рендер у хаба.
