## 0. Что такое ядро (дословно по user'у)
Хаб знает про все контейнеры (**инфа**) и умеет с ними говорить (**канал**), за единой дверью.
Ядро = три тонкие части: **Реестр · Дверь+лендинг · Канал**. Всё генерится из одного источника
(манифесты), инструменты (агенты/девопс/графики) — продукты, втягиваются поверх.

```
        ХАБ-ЯДРО (тонкое, одно на воркспейс, свой контейнер)
   ┌────────────────────────────────────────────────┐
   │ 1. РЕЕСТР   ← скан omnifield.yaml продуктов       │ инфа о контейнерах
   │ 2. ДВЕРЬ+ЛЕНДИНГ  gateway :8080, генерятся из     │ reach
   │    реестра (маршруты + карточки продуктов)        │
   │ 3. КАНАЛ    docker-ops: status/start-stop/exec    │ канал общения
   │    (старт — Portainer; свой — на роадмап)          │
   └────────────────────────────────────────────────┘
        ↑ продукты плагинятся: brainer(агенты)·devopser(девопс)·grafana
```

## 1. Закрытые развилки
- **Р1 — Канал: Portainer на старте, свой — на роадмап.** Portainer (opensource, уже `:9443`) даёт
  status/start-stop/exec из коробки + практика на container-API. **Свой control-plane неизбежен**
  (Portainer не покроёт спавн-агентов и кастомные ops) — заложен как след. инкремент, не «может быть».
- **Р2 — Реестр НА МАНИФЕСТАХ, без полумер.** Источник реестра = скан `omnifield.yaml` (инк1). НЕ
  рукописный `products.md` (тот **ретайрится** — становится выведенным отчётом/индексом, не
  источником). Продукт без манифеста просто не в реестре — корректно, не костыль. **Инк1 = жёсткий
  пререквизит** (см. §5).
- **Р3 — Хаб = отдельный тонкий контейнер** в compose воркспейса. Docker-доступ — ТОЛЬКО у hub-core
  и Portainer (продукты не получают socket — канон безопасности). devopser (продукт-девопс) собирает/
  генерит hub-core, но работает он как «хаб», не как devopser.

## 2. Реестр (сердце ядра)
- **Вход:** `omnifield.yaml` в корне каждого продукт-репо (контракт инк1, схема
  `@omnifield/contract-manifest`).
- **Процесс:** hub-core сканит воркспейс → валидирует каждый против `omnifield.schema.json` →
  собирает реестр. Невалидный/отсутствующий манифест = продукт не в реестре (loud-warn, не тихо).
- **Запись реестра** (per продукт): `name · type · title · description · routes[{path,port,service}]
  · integration{scopes,spawnEligible,deps} · liveStatus`. Статические поля — из манифеста; `liveStatus`
  (running/stopped) — из Канала (§4).
- **Единственный источник.** `registry/products.md` и `BRAINER_REPOS` — выводятся отсюда, не питают.

## 3. Дверь + лендинг (генерятся из реестра)
- **Маршруты gateway:** на каждый `reach.routes[]` → nginx `location <path> → <alias>:<port>` по
  паттерну этой сессии (docker-сеть, resolver, single-origin :8080). ⚠️ **Генерятся из реестра**, не
  руками — мой ручной `nginx.conf` = **референс-шаблон генератора** (это инк2).
- **Лендинг `/`:** генерится из реестра — карточка на продукт (title, description, ссылка на маршрут,
  индикатор running/stopped). Заменяет рукописный `hub/index.html`.
- Наружу — только `:8080` (канон [[single-origin-only-8080]]).

## 4. Канал (Portainer на старте)
- **Portainer** втягивается в хаб как «канал»: container list/status (→ `liveStatus` реестра),
  start/stop/restart, exec/console (**точечный вход** в контейнер). Доступен за дверью (линк с лендинга).
- **Убирает тебя-роутера:** `~/oa` / `docker start` / `docker network connect` / recreate — которые мы
  в этой сессии делали руками — становятся операциями Канала.
- **Свой control-plane (роадмап):** когда упрёмся в потолок Portainer (спавн-агентов от brainer,
  кастомные ops, тонкая интеграция с реестром) — пишем свой на docker-API. Практика на Portainer-API
  сейчас = разведка требований к своему.

## 5. Пререквизит и последовательность (честно, из Р2)
Ядро на манифестах ⇒ **инк1 обязан быть реальным ДО hub-core**:
1. **knowledger:** собрать `@omnifield/contract-manifest` (Zod-схема + эмит `omnifield.schema.json`).
   Дизайн готов (`knowledger/briefs/inc1-product-manifest-design.md`), **кода нет** — это первый шаг.
2. **product-owner'ы:** положить `omnifield.yaml` в корень (хотя бы 1 продукт для проверки петли —
   weber или brainer).
3. **devopser (hub-core):** скан манифестов → реестр → генерация лендинга + gateway-маршрутов;
   интеграция Portainer как Канала.

Продукты добавляются в хаб по мере появления манифеста — без переписывания ядра.

## 6. Физическая форма (Р3)
- **hub-core-сервис** (тонкий): scan+валидация манифестов → реестр (JSON/in-memory) → генерит
  `nginx.conf` + лендинг. Zero-heavy.
- **gateway (nginx)** — дверь, ест сгенерированный конфиг.
- **Portainer** — Канал.
- Всё три — контейнеры в compose воркспейса; docker-socket — только hub-core (для liveStatus, если не
  через Portainer-API) и Portainer. Продукты — никогда.
- Дом workspace-архитектора — этот хаб-контейнер (снимает «где живёт координатор», блюпринт §5/§7).

## 7. Первый строимый срез (после инк1)
**hub-core v0 = Реестр(скан манифестов) → генерация лендинга + gateway-маршрутов; Канал = Portainer-линк.**
DoD: положил `omnifield.yaml` в продукт → он появился карточкой на `:8080` с живым статусом, маршрут
работает, container-ops доступны через Канал. Ни одной рукописной строки в nginx/лендинге/products.md.

## 8. Границы / не-цели
- Ядро тонкое: реестр+дверь+канал. Логика продуктов (агенты/девопс/графики) — в продуктах, втягивается.
- Свой control-plane — НЕ сейчас (Portainer), но заложен (Р1).
- Никаких полумер (Р2): реестр не питается рукописным products.md даже временно.
