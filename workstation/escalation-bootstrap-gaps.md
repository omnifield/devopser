## Дыра 1 — install-пути кода НИКОГДА не исполнялись (главное)

На тачке все 6 тулзов уже стоят → `-Verify` и обычный запуск уходят в ветку
«base layer already complete». Поэтому **ни разу не отработали**:
- реальный `winget install ...` (5 тулзов),
- нативный installer claude (`Invoke-RestMethod claude.ai/install.ps1 | iex`),
- `Update-SessionPath` после установки (свежий PATH из реестра),
- ветки отчёта `INSTALLED` / `FAILED` / `OUTDATED`.

DoD (§Verify, п.3) прямо требует **e2e на чистой машине**: bootstrap → clone по
repos.md → `uv sync` + `pnpm install` с нуля. **Этого прогона не было.** Локальный
`-Verify` его НЕ заменяет — он лишь читает уже-настроенную машину.

**Нужно:** прогон на реальной чистой тачке (лучший вариант — та самая вторая
машина из инцидента 2026-07-08). Owner без неё закрыть не может.

## Дыра 2 — идемпотентность проверена только частично

DoD п.2: «повторный запуск — все skip, ничего не ломает». Сейчас подтверждена
только ветка «уже всё стоит → early return». Полный цикл
**install → повторный запуск → skip** не прогонялся (следствие дыры 1).

## Дыра 3 — расхождение «10.x» vs фактические 11.x (мелочь, но зафиксировать)

Амендмент №2 §2: «bootstrap ставит «любой 10.x»». Фактически winget `pnpm.pnpm`
сейчас отдаёт 11.3.0. Код это принимает корректно (`MinMajor ≥ 10`, пин
`packageManager` в репо переключит на нужную версию сам) — семантика верна.
Но текст брифа/README вводит в заблуждение. **Решение architect:** поправить
формулировку на «любой ≥10» или осознанно запинить мажор.

## Предложение

1. Выделить чистую тачку (или VM/sandbox) под e2e-прогон — закрывает дыры 1 и 2 разом.
2. По результату — либо «DoD зелёный», либо баг-лист по фактическим install-путям.
3. Параллельно — микрофикс формулировки «10.x → ≥10» (дыра 3), это owner может
   сделать сразу по отдельному указанию.
